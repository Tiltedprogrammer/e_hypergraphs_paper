\section*{Section}

Slotted e-graphs~\cite{slotted-egraphs} are an extension of e-graphs with two main objectives (apart from representing sets of equivalent terms)
\begin{itemize}
  \item Uniquely represent terms that are different up to a `renaming' of free variables, i.e., terms $a + b$ and $c + d$ where $a,b,c,d$ are distinct variables are represented as the same e-node.
  \item Uniquely represent terms that are different up to renaming of bound variables, i.e., terms $\text{bind}\;\$s_1\; t_1$ and $\text{bind}\; \$s_2\;t_2$ are represented by the same e-node if $(s_1 \; s') \text{bind}\; \$s_1\;t_1 = (s_2 \; s') \text{bind}\;\$s_2\;t_2$ where $s'$ is a fresh name.
\end{itemize}

The first is achieved by parameterising each e-class by the ordered set of free variables (\textit{slots}) the terms of the e-class contain.
For example, a term $a + b$ contains two free variables $a$ and $b$ and therefore the e-class that contains this term is parameterised by two variables.
Non duplication of $a + b$ and $c + d$ is achieved by computing normal forms (\textit{shapes}) of such terms.
In this particular example the shape is $\$0 + \$1$.
One of the components of a slotted e-graph is a hash-map that maps such shapes to e-class id's that contain equivalent terms.
Since two terms evaluate to the same shape, they correspond to a singe entry in an e-graph.

The other one is achieved by both computing the shape and generating a fresh slot each time a term with a binder is added to an e-graph.
This makes it impossible to write a capturing $\beta$-reduction such as $(\lambda x . \lambda y . x) y$.

Both of the above objectives can be achieved by using string diagrams for Cartesian closed SMCs by essentially performing a closure conversion of every functional symbol.

Consider the example in Figure~\ref{fig:slotted_a_b_c_d} where subfigure (c) represents a term $(a + b) + (c + d)$ and note how subterms for $(a+b)$ and $(c + d)$ are shared.
The headers of e-classes specify which free variables parameterise the e-class.
Subfigure (d) represents an e-graph after an equivalent term $a + (b + (c + d))$ was added.

\begin{figure}

  \caption{Slotted e-graph example~\cite{slotted-egraphs}}
  \label{fig:slotted_a_b_c_d}
\end{figure}

We can express the same data using string diagrams as in Figure~\ref{fig:slotted_a_b_c_d_string_1}.
Note that there is no 1-to-1 corresponding between the slotted e-graph and the resulting string diagram (unlike in Figure~\ref{fig:slotted_a_b_c_d_string_2} which can not be obtained by applying rewrite rules to the initial string diagram).
However, the e-box in red roughly corresponds to the invocations $(a)\;(b,c,d)$ and $(a,b)\;(c,d)$ of the slotted e-graph.
We can decompose a string diagram in such a way because of the copy-delete rule.
Also note that we can technically achieve more sharing by not only doing closure conversion on subtrees but on subgraphs as well, i.e., we could share the $\lambda x y . x + y$ 3 times instead of 2.

In the presence of $\beta$-reduction rule we need a richer structure for our string diagrams though.
Consider a term $(\lambda\; x\; .\; f\;x)\; y$ which evaluates to $f \; y$ under $\beta$-reduction.
The corresponding slotted e-graph (after a $\beta$-reduction has been applied) can be seen in Figure~\ref{fig:f_x_y_slotted}.
Note that it contains a cycle (the rightmost \texttt{app} e-node references \texttt{bind} and the latter references the e-class containing \texttt{app} again).
This e-graph contains an infinite family of terms: $(\lambda\; x\; .\; f\;x)\; y$, $f \; y$, $(\lambda\; x\; .\; (\lambda z\; .\; f\;z) x\; y$ and so on.
To fully support this, we need a trace.

\subsection{Shape}

A \textit{shape} is a pair $(e, \sigma)$ where $e$ is an e-node and $\sigma$ is a slot map for this particular e-node.
The e-node $e$ can be thought of a canonicalized version of the e-node $\sigma(e)$ (i.e. slot map applied to $e$). The \textit{hashcons} data structure maps such canonicalized e-nodes to e-class ids.
This is how non-duplication is maintained.
Below is the code for computing the shape.
In plain words, it takes an e-node $c$ and replaces all the slots in $c$ with canonical numerical slots $\$1, \$2$ etc..
For example, given an e-node \mintinline{rust}{Var(Slot::named("x"))} (a code for $\texttt{var \$x}$ e-node), the canonicalized version will be $\texttt{var \$1}$ (or, \mintinline{rust}{Var(Slot::numeric(1))}).
Now, to give a more complex example, suppose we were to add $\lambda x . x$ and $\lambda y . y$ expressions to a slotted e-graph.
In code this would be 
\begin{minted}{rust}
  Lambda::Lam(Bind { 
    slot: Slot::named("x"),
    elem: AppliedId { id: Id(0), m: {"f0" -> "x"}}
  })
\end{minted}

Where \mintinline{rust}{AppliedId { id: Id(0), m: {"f0" -> "x"}}} is the e-class id for the variable $x$ inside $\lambda x . x$.
We get the same e-node for the $\lambda y . y$ with "x" replaced by "y".
The slots of this e-node are the two occurrences of slot $x$ (one for the binder and another one from the values of the \mintinline{rust}{AppliedId}).
Therefore, the first component of the shape will be

\begin{minted}{rust}
  Lambda::Lam(Bind { 
    slot: Slot::numeric(1),
    elem: AppliedId { id: Id(0), m: {"f0" -> 1}}
  })
\end{minted}

and we get the same shape for $\lambda y . y$.


\begin{listing}
  \begin{minted}[linenos, fontsize=\small, breaklines]{rust}
    fn weak_shape(&self) -> (Self, Bijection) {
      let mut c = self.clone();
      let mut m = SlotMap::new();
      let mut i = 0;

      for x in c.all_slot_occurrences_mut() {
          let x_val = *x;
          if !m.contains_key(x_val) {
              let new_slot = Slot::numeric(i);
              i += 1;

              m.insert(x_val, new_slot);
          }

          *x = m[x_val];
      }

      let m = m.inverse();

      let public = c.slots();
      let m: SlotMap = m.iter().filter(|(x, _)| public.contains(x)).collect();

      (c, m)
  }
  \end{minted}
\end{listing}

\subsection{E-matching}

E-matching is a function $match(p,m * a, \beta, m_p)$ that takes in a pattern $p$, an applied-id $m*a$ to match against, a subtitution from variables to applied-ids $\beta$ and a renaming $m_p$ that maps slots of an e-graph to slots of the pattern.


\begin{listing}
\begin{minted}[linenos, fontsize=\small, breaklines]{rust}
  struct State {
    // uses egraph slots.
    partial_subst: Subst, // type Subst = HashMap<String, AppliedId>

    // maps from the pattern slots to the e-graph slots.
    partial_slotmap: SlotMap, //type SlotMap = HashMap<Slot,Slot>
  }

  pub fn ematch_all<L: Language, N: Analysis<L>>(
    eg: &EGraph<L, N>,
    pattern: &Pattern<L>,
) -> Vec<Subst> {
    let mut out = Vec::new();
    for i in eg.ids() {
        let i = eg.mk_sem_identity_applied_id(i); // applies identity renaming to an id to turn it into AppliedId
        out.extend(
            ematch_impl(pattern, State::default(), i, eg) // this is `match'
                .into_iter()
                .map(final_subst),
        );
    }
    out
}

\end{minted}
\end{listing}


\begin{listing}
\begin{minted}[linenos, fontsize=\small, breaklines]{rust}
fn ematch_impl<L: Language, N: Analysis<L>>(
    pattern: &Pattern<L>,
    st: State,
    i: AppliedId,
    eg: &EGraph<L, N>,
) -> Vec<State> {
    match &pattern {
        Pattern::PVar(v) => {
            let mut st = st;
            if let Some(j) = st.partial_subst.get(v) {
                if !eg.eq(&i, j) {
                    return Vec::new();
                }
            } else {
                st.partial_subst.insert(v.clone(), i);
            }
            vec![st]
        }
        Pattern::ENode(n, children) => {
            let mut out = Vec::new();
            for nn in eg.enodes_applied(&i) { //traverses e_nodes of the e-class
                let d = std::mem::discriminant(n);
                let dd = std::mem::discriminant(&nn);
                if d != dd {
                    continue;
                };

                //takes the e_node that matches 'n'.
                ematch_node(&st, eg, &n, children, &mut out, &nn);
            }
            out
        }
        Pattern::Subst(..) => panic!(),
    }
}

\end{minted}
\end{listing}

\begin{listing}
\begin{minted}[linenos, fontsize=\small, breaklines]{rust}
fn ematch_node<L: Language, N: Analysis<L>>(
    st: &State,
    eg: &EGraph<L, N>,
    n: &L, // from the pattern
    children: &[Pattern<L>],
    out: &mut Vec<State>,
    nn: &L, //candidate e_node from e-class
) {
    'nodeloop: for n2 in eg.get_group_compatible_weak_variants(&nn) {
        if CHECKS {
            assert_eq!(&nullify_app_ids(n), n);
        }

        let clear_n2 = nullify_app_ids(&n2);
        // We can use weak_shape here, as the inputs are nullified
        // i.e. they only have id0() without slot args, so there are no permutations possible.
        let (n_sh, _) = n.weak_shape();
        let (clear_n2_sh, _) = clear_n2.weak_shape();
        if n_sh != clear_n2_sh {
            continue 'nodeloop;
        } //checks if the shapes are equal, for example lambda x . f and lambda y . f have the same shape

        let mut st = st.clone();

        for (x, y) in clear_n2
            .all_slot_occurrences()
            .into_iter()
            .zip(n.all_slot_occurrences().into_iter())
        {
          // checks if mapping m_p is not ambiguous
            if !try_insert_compatible_slotmap_bij(x, y, &mut st.partial_slotmap) {
                continue 'nodeloop;
            }
        }

        let mut acc = vec![st];
        // recursively matches children
        for (sub_id, sub_pat) in n2.applied_id_occurrences().into_iter().zip(children.iter()) {
            let mut next = Vec::new();
            for a in acc {
                next.extend(ematch_impl(sub_pat, a, sub_id.clone(), eg));
            }
            acc = next;
        }

        out.extend(acc);
    }
}

\end{minted}
\end{listing}

\section{Explicit substitution}

Lambda calculus with explicit substitution (also $\lambda_{\sigma}$-calculus) made the substitution operator a first-class element of the calculus (rather than a meta-level operation) to bridge the gap between lambda calculus and its concrete implementations~\cite{AbadiExplicitSubst}.
However, the existence of unique normal forms turned out to be a delicate matter and to achieve uniqueness, lambda terms must be quotiented by \textit{graphical} equivalence in addition to $\alpha$-equivalence~\cite{Standardisation}.
We claim that using e-hypergraphs over e-graphs (or slotted e-graphs) may be more beneficial for doing equational reasoning for lambda calculus with explicit substitution.

The rules can be seen in a table below (plust a standard rule for $(\lambda x . u)v = u[x / v]$)

\begin{tabular}{lcc}
\texttt{beta} & $(\lambda x . t) u = t[x / u]$&\\
\texttt{subst\_same\_var} & $x[x / u] = u$ &\\
\texttt{subst\_diff\_var} & $y[x / u] = y$ & if $x \not \in \mathcal{F}(u)$\\
\texttt{subst\_application} & $(uu')[x / v] = u[x / v]u'[x / v]$ & \\
\texttt{subst\_lambda} & $(\lambda x . u)[y / v] = \lambda x . (u[y / v])$ & if $x \not \in \mathcal{F}(v)$ \\
\texttt{subst\_interchange\_1} & $(u[x / v])[x' / v'] = (u[x' / v'])[x / v[x' / v']]$ & if $x' \in \mathcal{F}(v)$ and $x \in \mathcal{F}(v')$\\
\texttt{subst\_interchange\_2} & $(u[x / v])[x' / v'] = (u[x / v[x' / v']])$ & if $x' \not \in \mathcal{F}(u)$\\
\texttt{subst\_comm} & $(u[x / v])[x' / v'] = (u[x' / v'])[x / v]$ & if $x \not \in \mathcal{F}(v')$ and $x' \not \in \mathcal{F}(v)$\\
\end{tabular}

while such rules are absorbed when using string diagrams (or hypergraphs) akin to proof nets, they are not absorbed by e-graph representation.
In Table~\ref{tbl:slotted} one could see the number of e-nodes for different set of equations enabled for saturating an expression \texttt{plus 2 2} using Church encoding.

\begin{table}
\begin{tabular}{lccccc}
  Set of equations & built-in subst & explicit subst & explicit subst & explicit subst & explicit subst\\
  &&& + comm& + interchange&  + interchange\\
  &&&&&  + comm\\
  \# of e-nodes & 40 & 161 & 288 & 289 & 356
\end{tabular}
\caption{Slotted e-graphs benchmarks}
\label{tbl:slotted}
\end{table}

String diagrams represent the same equivalence without loss of generality and without handling equations for substitutions explicitly.

Recall that by the virtue of free enrichment we have the equality $\Lambda(f + g) = \Lambda(f) + \Lambda(g)$.
The intuitive interpretation of this equality is that $f \sim g$ (due to rewrite rules being applied) implies $\Lambda(f) \sim \Lambda(g)$.
This is exactly one of the equivalences that a slotted e-graph from~\cite{slotted-egraphs} maintains.

\[
\text{bind}\;\$s_1\; x \cong \text{bind}\;\$s_2\;y \;\; \text{if} (\$s_1 \$f) * x \cong (\$s_2 \$f) * y, \text{where $f$ is fresh}
\]

\begin{remark}

The rules arise from linear substitution lambda calculus which is as follows

\begin{align*}
(\lambda\;x\;t)\;L\;u \mapsto_{db} t[x / u]L\\
C\llbracket x \rrbracket [x / u] \mapsto_{ls} C \llbracket u \rrbracket [x / u]\\
t[x/u] \mapsto t, \text{if } x \not \in fv(t)
\end{align*}

The contexts $C$ are defined as

\[
C \Coloneqq \square \;|\; \lambda x . C \; | \; C\;t \;|\; t\;C\;|\;C[x / t]\;|\;t[x / C]
\]

and in particular in $\lambda x . C$ and $C[x/t]$ $x$ is bound.

\end{remark}

\section{EggLog}

This is a Datalog-like implementation of equality saturation.
In this representation there is no notion of an e-graph \textit{per se}, but rather a bunch of relations (or partial functions) and rules.
The motivation stems from Datalog implementation of finding the shortest paths in a graph.
Consider a program below

\begin{minted}[linenos]{prolog}
(function edge (i64 i64) i64)
(function path (i64 i64) i64 :merge (min old new)) 

(rule ((= (edge x y) len)) 
((set (path x y) len))) 
(rule ((= (path x y) xy) 
(= (edge y z) yz))
((set (path x z) (+ xy yz))))

(set (edge 1 2) 10)
(set (edge 2 3) 10) 
(set (edge 1 3) 30)

(run) 14 (check (path 1 3)) ;; prints "20"  
\end{minted}

In the above, the function \mintinline{prolog}{edge} specifies a function that for two vertices (represented with numbers) assigns a weight.
Similarly, the function \mintinline{prolog}{path} assigns a weight to a path.
\mintinline{prolog}{set} essentially defines a function for provided arguments.

\texttt{egglog} extends the above machinery with custom \textit{datatypes} and \textit{union'ing} of ids which are returned by constructors for each variant of a datatype.
Datalog's semantics is iterative application of rules until a fixed point is reached. The \texttt{merge}, \texttt{union} and \texttt{canonicalization} are backed inside the engine.
Semantics of \texttt{egglog} is given below.


% ================================================================
% OVERVIEW
% ================================================================

\paragraph{Intuition.}
Core \texttt{egglog} is a simplified, mathematically clean subset of
the full \texttt{egglog} language.
It defines precisely what a program looks like (syntax)
and what it means to run one (semantics).
Compared to full \texttt{egglog}, it restricts:
\begin{itemize}
  \item Each rule may only have one head atom.
  \item There is no \texttt{union} operator.
  \item The meaning of \texttt{:merge} is fixed:
    for IDs (uninterpreted constants) it is \texttt{union},
    and for interpreted constants it is the lattice join~$\sqcup$.
\end{itemize}
These restrictions make the semantics easier to reason about
while preserving the core behavior of \texttt{egglog}.

% ================================================================
% SYNTAX
% ================================================================

\subsection{Syntax}

Given the set of (interpreted) constants~$C$,
the syntax of core \texttt{egglog} is shown below in~\cref{fig:syntax}.

\begin{figure}[h]
  \centering
  \begin{tabular}{llcl}
     Program        & $P$ & ::= & $R_1,\ldots, R_n$ \\
     Rule           & $R$ & ::= & $A \gets A_1,\ldots A_m$ \\
     Atom           & $A$ & ::= & $f(p_1,\ldots, p_k)\mapsto o
                                  \mid f(p_1,\ldots, p_k)$ \\
     Pattern        & $p$ & ::= & $f(p_1,\ldots, p_k)\mid o$ \\
     Term           & $t$ & ::= & $f(t_1,\ldots, t_k)\mid v$ \\
     Base pattern   & $o$ & ::= & $v\mid x$ \\
     Constant       & $v$ & ::= & $c\mid n$ \\
     Interpreted Constant & $c\in C$ & & (e.g. numbers, booleans)\\
     Uninterpreted Constant & $n\in N$ & & (e.g. e-class ids)\\
     Variable       & $x, y, \ldots$ &
  \end{tabular}
  \caption{Syntax of core \texttt{egglog}.}
  \label{fig:syntax}
\end{figure}

\paragraph{Explanation.}
A program is a list of rules.
Each rule has a \emph{head atom} (what we want to derive)
and a list of \emph{body atoms} (the facts that justify it).

An atom such as
\[
  f(p_1, \ldots, p_k) \mapsto o
\]
intuitively means that function~$f$,
when applied to arguments~$p_1,\ldots,p_k$,
has value~$o$.

\begin{itemize}
  \item A \emph{pattern} may contain variables and nested expressions.
  \item A \emph{term} is a pattern with no variables (i.e. ground).
  \item An \emph{atom} is ground if all its patterns are ground.
  \item \emph{Interpreted constants}~$c$ have semantic meaning
        (e.g. numeric values);
        \emph{uninterpreted constants}~$n$ are symbolic identifiers.
\end{itemize}


% ================================================================
% SEMANTICS OVERVIEW
% ================================================================

\subsection{Semantics Overview}

\paragraph{Intuition.}
An \texttt{egglog} program maintains a database of known facts
and equivalences between constants,
and repeatedly applies rules to derive new facts.
Two main components define the semantics:

\begin{enumerate}
  \item The \emph{immediate consequence operator}~$T_P^\uparrow$,
        which applies rules once.
  \item The \emph{rebuilding operator}~$R$,
        which restores consistency by merging equal terms
        and joining lattice values.
\end{enumerate}

% ================================================================
% INSTANCES AND DATABASES
% ================================================================

\paragraph{Instances.}
We assume:
\begin{itemize}
  \item An infinite set of uninterpreted constants
        $N = \{n_1, n_2, \ldots\}$.
  \item A complete lattice $L = (C, \sqsubseteq, \sqcup)$
        over interpreted constants~$C$,
        with least element~$\bot$.
\end{itemize}

A \emph{schema}~$S$ assigns function symbols types over $\{N, C\}$.
Given a schema, an \emph{instance}
\[
  I = (\DB, \equiv)
\]
consists of:
\begin{itemize}
  \item A database $\DB$ of function entries
        $f(v_1,\ldots,v_k)\mapsto v$;
  \item An equivalence relation~$\equiv$ over $N\cup C$.
\end{itemize}

\paragraph{Remark.}
The equivalence relation $\equiv$
encodes what is known to be equal
(similar to E-class IDs in equality saturation).
Interpreted constants (like numbers) are only equal to themselves.


% ================================================================
% JUDGMENT: WHAT FACTS HOLD
% ================================================================

\paragraph{Judgment.}
We write $I \vdash A$ to mean ``the ground atom~$A$ is contained in~$I$.''
Formally:

\begin{mathpar}
    \inferrule{
        I\vdash t_i\mapsto v_i \text{ for }i=1\ldots k \\
        I=(\DB, \equiv)\quad
        f(v_1, \ldots, v_k)\mapsto v \in \DB
    }{
        I\vdash f(t_1, \ldots, t_k)\mapsto v
    }
    \and
    \inferrule{}{I\vdash v\mapsto v}
    \and
    \inferrule{
        I\vdash f(t_1, \ldots, t_k)\mapsto v
    }{
        I\vdash f(t_1, \ldots, t_k)
    }
\end{mathpar}

\paragraph{Explanation.}
The first rule says:
if all arguments of $f$ evaluate to some values~$v_i$
and $\DB$ contains $f(v_1,\ldots,v_k)\mapsto v$,
then we know $f(t_1,\ldots,t_k)\mapsto v$.

The second rule ensures that any constant trivially maps to itself.
The third allows us to ignore the output when only existence matters.


% ================================================================
% FLATTENING
% ================================================================

\paragraph{Flattening.}
Nested atoms such as
$f(g(1),2)\mapsto 3$
must be flattened into simpler entries
before insertion into~$\DB$.
Flattening introduces auxiliary IDs for subterms.

\begin{figure}[h]
\begin{align*}
    \flatten_I(A) &= s \qquad \text{where } (v, s) = \aux(A)&\\
    \aux(f(t_1,\ldots,t_k)\mapsto v)
      &= \bigl(v,\ \{f(v_1,\ldots,v_k)\mapsto v\}
                   \cup \bigcup_i s_i\bigr)\\
      &\text{where } (v_i,s_i)=\aux(t_i)\\[1ex]
    \aux(f(t_1,\ldots,t_k))
      &= \bigl(v,\ \{f(v_1,\ldots,v_k)\mapsto v\}
                   \cup \bigcup_i s_i\bigr)\\
      &\text{where } (v_i,s_i)=\aux(t_i)
         \text{ and }
         v=\begin{cases}
              \text{existing value of } f(v_1,\ldots,v_k),\\
              \text{default}_f \text{ otherwise.}
            \end{cases}\\
    \aux(v)&=(v,\emptyset)
\end{align*}
\caption{Flattening nested atoms before insertion into the database.}
\end{figure}

\paragraph{Example.}
Flattening
$f(g(1),2)\mapsto 3$
yields:
\[
\{\, g(1)\mapsto n_1,\ f(n_1,2)\mapsto 3 \,\}
\]
where $n_1$ is a fresh ID.
If the output type of $f$ is~$N$, $\textit{default}_f$ is a fresh~$n$;
otherwise it is the lattice bottom~$\bot$.

% ================================================================
% IMMEDIATE CONSEQUENCE OPERATOR
% ================================================================

\subsection{Immediate Consequence Operator}

\paragraph{Intuition.}
This operator applies all rules once to derive new facts,
similar to Datalog’s $T_P$ operator,
but extended with flattening and non-monotone support.

Formally, for program~$P$ and instance~$I=(\DB,\equiv)$:
\[
  T_P^\uparrow(I) = \DB \cup T_P(I)
\]
where
\[
  T_P(I) = (\DB', \equiv), \quad
  \DB' = 
    \bigcup_{(A\gets A_1,\ldots,A_m)\in P}
    \{\, \flatten_I(A[\sigma]) \mid
         \forall_i\ I\vdash A_i[\sigma] \,\}.
\]

\paragraph{Explanation.}
For each rule, we consider all substitutions~$\sigma$
mapping variables to constants.
If all body atoms $A_i[\sigma]$ are true in the current database,
we add the flattened head $A[\sigma]$ to the database.

The union with the old $\DB$ makes this operator
\emph{inflationary}—facts are only added, not removed.

% ================================================================
% REBUILDING OPERATOR
% ================================================================

\subsection{Rebuilding Operator}

\paragraph{Intuition.}
After applying rules, functions may map the same key to
different outputs.
The rebuilding operator~$R$ repairs this by merging
equal IDs and joining lattice values.

\begin{align*}
  R((\DB,\equiv)) &= (\DB_R, \equiv_R)\\[1ex]
  \equiv_R &=
    \text{equivalence closure of }
    \bigl(
      \equiv \cup
      \{(n_1,n_2)\mid
        f(v_1,\ldots,v_k)\mapsto n_1\in\DB,\
        f(v_1,\ldots,v_k)\mapsto n_2\in\DB,\
        n_1,n_2\in N\}
    \bigr)\\[1ex]
  \DB_R &=
    \lambda_{\equiv_R}\!\left(
      \{\,f(v_1,\ldots,v_k)\mapsto
          \mergef_{f,\equiv}(K)
          \mid
          K=\{v\mid f(v_1,\ldots,v_k)\mapsto v\in\DB\},
          K\neq\emptyset
      \}
    \right)
\end{align*}

where
\[
\mergef_{f,\equiv}(K)=
\begin{cases}
  \min(\lambda_\equiv(K)) & \text{if output type of $f$ is $N$;}\\
  \bigsqcup K              & \text{if output type of $f$ is $C$.}
\end{cases}
\]

\paragraph{Explanation.}
\begin{itemize}
  \item If $f$ maps the same inputs to multiple
        uninterpreted outputs $n_1,n_2$,
        we record $n_1\equiv n_2$.
  \item If it maps to multiple interpreted outputs,
        we take their lattice join $\bigsqcup K$.
  \item We then \emph{canonicalize} all entries
        so that each equivalence class has a single representative.
\end{itemize}

Since canonicalization can introduce new equalities,
we iterate rebuilding to a fixpoint $R^\infty(I)$.

% ================================================================
% FULL STEP AND FIXPOINT SEMANTICS
% ================================================================

\subsection{Program Evaluation}

\paragraph{One iteration.}
One full iteration of evaluation combines rule application
and rebuilding:
\[
  F_P = R^\infty \circ T_P^\uparrow.
\]
Applying $F_P$ once means:
(1) apply all rules to generate new facts,
(2) rebuild until consistent.

\paragraph{Monotonic growth.}
Each iteration yields a database that ``knows more.''
Define the \emph{expanded database}~$E_\equiv(\DB)$
to include all consequences of equivalence and lattice order.
Then we can order instances $(\DB_1,\equiv_1)\sqsubseteq_I(\DB_2,\equiv_2)$
iff
\[
E_{\equiv_1}(\DB_1)\subseteq E_{\equiv_2}(\DB_2)
\quad\text{and}\quad
\equiv_1\subseteq\equiv_2.
\]
Even though $F_P$ need not be monotone itself,
the sequence
\[
I_\bot
\sqsubseteq_I F_P(I_\bot)
\sqsubseteq_I F_P(F_P(I_\bot))
\sqsubseteq_I \cdots
\]
is always increasing,
where $I_\bot=(\emptyset,\textit{Id}_{N\cup C})$
is the empty initial database.

\paragraph{Fixpoint semantics.}
The meaning of a program~$P$ is the inductive fixpoint:
\[
  \llbracket P \rrbracket = F_P^\infty(I_\bot).
\]
In practice, implementations approximate this by
running a finite number of iterations
$(R^\infty\circ T_P^\uparrow)^n(I_\bot)$.

\paragraph{Summary.}
\begin{itemize}
  \item \textbf{Atoms} represent facts about functions.
  \item \textbf{Instance $(\DB,\equiv)$} stores all known facts and equalities.
  \item \textbf{Flattening} normalizes nested terms.
  \item \textbf{$T_P^\uparrow$} applies all rules once.
  \item \textbf{$R$} rebuilds to maintain functional consistency.
  \item \textbf{$F_P$} performs one full reasoning step.
  \item \textbf{Fixpoint} of $F_P$ gives the final meaning of the program.
\end{itemize}
